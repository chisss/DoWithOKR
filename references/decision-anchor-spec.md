# 决策锚点与偏离检测规范

本文档定义 DoWithOKR 的决策锚点机制和偏离检测流程，确保角色在执行过程中始终对齐 KR 验收标准。

## 设计理念

角色"自主决定实现路径"不等于"随意执行"。每个技术决策都必须能回答一个问题：**"这个选择是否有助于达成我的 KR 验收标准？"** 如果无法回答，该决策属于范围蔓延。

## 决策锚点

### 四类决策类型

| 决策类型 | 锚点问题 | 示例 |
| --- | --- | --- |
| 技术选型 | 哪个方案更有利于达成 KR 的质量指标？ | BE 选择 ORM 框架时，锚定 KR 中的"响应时间 < 200ms" |
| 范围判断 | 这个工作是否在我的 KR 范围内？ | FE 发现后端 API 有 bug，不自己修，标记为阻塞 |
| 质量取舍 | 当前质量是否满足 KR 的量化标准？ | QA 发现测试覆盖率 82%，KR 要求 ≥ 85%，继续补充 |
| 时间权衡 | 是否在 KR 的时间约束内？ | BE 评估重构需要 3 天，但 KR 截止日期只剩 2 天，选择最小可行方案 |

### 锚点输出格式

角色在执行报告中记录关键决策时，使用以下格式：

```
决策: 选择 {方案}
锚定: {KR-ID} 的验收标准 "{具体标准}"
理由: {为什么这个方案更有利于达成该标准}
```

### 使用规则

- 不是每个微小操作都需要锚点，只在**关键决策点**（技术选型、范围边界、质量取舍、时间权衡）输出锚点。
- 如果某个决策同时服务于多个 KR，列出主要的一个即可。
- 锚点是角色的自我约束工具，不是额外的审批流程。

## 偏离检测

### 触发条件

满足以下任一条件即触发偏离警告：

1. **范围蔓延**：角色正在做的工作无法关联到任何 KR 验收标准
2. **越界操作**：角色正在处理其他角色职责范围内的问题
3. **时间超限**：角色的工作时间已超过 KR 时间约束

### 检测方式

角色在执行每个主要操作前，自问："这个操作对应哪个验收标准？"

- 能回答 → 继续执行
- 无法回答 → 触发偏离警告

### 偏离警告格式

```markdown
⚠️ 偏离检测
- 当前操作: {操作描述}
- 关联 KR: 无法关联
- 偏离类型: 范围蔓延 / 越界操作 / 时间超限
- 建议: {处理建议}
```

### 处理规则

| 偏离类型 | 处理方式 |
| --- | --- |
| 范围蔓延 | 停止操作，回到 KR 验收标准重新规划 |
| 越界操作 | 标记为阻塞，记录到 status.md，建议由对应角色处理 |
| 时间超限 | 评估最小可行交付，与用户确认是否降级 KR 标准 |
