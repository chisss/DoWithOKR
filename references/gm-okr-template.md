# GM OKR Template

## 甲方需求

用一句话复述用户的自然语言需求，并标记已确认与待确认内容。

- **已确认**：用户明确表达的功能、范围、约束
- **待确认**：模糊表述、隐含假设、未提及但可能影响交付的内容

## GM Objective

以甲方视角表达本次需求要达成的业务或产品结果。Objective 应当：
- 方向明确，一句话概括最终交付价值
- 不包含具体技术方案或实现细节
- 可被非技术人员理解

### KR 质量检查清单（SMART 原则）

- [ ] **Specific（具体）**：KR 描述了一个明确的可交付物或可观测的结果
- [ ] **Measurable（可衡量）**：有清晰的验收标准，能判断"做到了"还是"没做到"
- [ ] **Achievable（可达成）**：在当前资源和时间约束下可以完成
- [ ] **Relevant（相关）**：直接支撑 GM Objective，删掉它会影响目标达成
- [ ] **Time-bound（有时限）**：属于当前交付周期，不是"未来某天"

## 边界

列出本轮明确不做的内容，避免范围蔓延。

### 边界判断规则

- **纳入**：用户明确要求 + 直接影响 Objective 达成的内容
- **排除**：用户未提及且不影响核心交付的内容；"锦上添花"类优化
- **待定**：用户提及但描述模糊，需确认后再决定纳入或排除

## 待确认

列出需要用户确认的问题；确认前不得把不确定内容标记为已完成。

## 常见反模式

| 反模式 | 说明 | 修正方式 |
| --- | --- | --- |
| KR 太模糊 | "提升用户体验" — 无法验证 | 改为可观测结果："首页加载时间 < 2s" |
| KR 是任务 | "编写单元测试" — 这是活动不是结果 | 改为结果："核心模块测试覆盖率 ≥ 80%" |
| KR 不可证伪 | "代码质量良好" — 没有判定标准 | 加入量化指标或具体验收条件 |
| KR 超出周期 | 单个 KR 需要多个迭代才能完成 | 拆分为当前周期可交付的子结果 |
| KR 与 O 脱节 | KR 完成了但 Objective 没有推进 | 重新审视 KR 是否真正支撑 Objective |

## 示例：一个合格的 GM OKR

**甲方需求**：为内部管理系统增加用户权限模块，支持角色分配和菜单控制。

**GM Objective**：交付可用的 RBAC 权限管理模块，使管理员能按角色控制用户的菜单和操作权限。

| KR ID | 验收标准 | 证据要求 | 状态 |
| --- | --- | --- | --- |
| GM-KR1 | 管理员可创建角色并分配菜单权限，保存后立即生效 | 操作录屏 + 数据库记录截图 | 未开始 |
| GM-KR2 | 用户登录后仅看到其角色对应的菜单项，无越权访问 | 不同角色登录对比截图 + 接口测试报告 | 未开始 |
| GM-KR3 | 权限变更不需要重启服务，热生效延迟 < 5s | 变更前后对比日志 | 未开始 |
