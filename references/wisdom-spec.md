# 角色能力积累规范（Wisdom）

本文档定义 DoWithOKR 的角色能力积累机制，包括存储结构、文件格式、读写时机和各角色记忆侧重。

## 设计理念

OKR 的真正价值不仅在于单次交付，更在于通过反复的目标-执行-回顾循环，积累组织能力。角色不是无状态的执行器，而是有经验的专业人士。

## 目录结构

```text
.okr/wisdom/
  gm.md        # GM 的能力积累
  pd.md        # PD 的能力积累
  archd.md     # ArchD 的能力积累
  be.md        # BE 的能力积累
  fe.md        # FE 的能力积累
  qa.md        # QA 的能力积累
  team.md      # 团队级别的组织智慧
```

文件按需创建：角色首次参与 OKR 周期并完成评分后，由 `okr-review-score` 自动创建对应的 wisdom 文件。

## 文件格式

```markdown
---
role: GM
updated: 2026-04-30
cycles_completed: 3
---

## 专业知识

- 需求拆解时，优先识别"用户价值"而非"技术实现"
- KR 中包含"覆盖率"指标时，团队执行效果显著提升

## 项目上下文

- SchemaPlex 技术栈: Spring Boot 3 + MyBatis-Plus + Vue 3
- 核心瓶颈: Agent 并发执行的资源隔离

## 反思记录

### Cycle 1 (2026-04-15)
- 教训: KR "完成 Agent 开发" 太宽泛，评分时无法客观判断
- 改进: 后续 KR 必须包含量化指标
- 效果: Cycle 2 评分争议减少 80%
```

### frontmatter 字段

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| role | string | 角色缩写（GM、PD、ArchD、BE、FE、QA 等），team.md 使用 `team` |
| updated | string | 最后更新日期 YYYY-MM-DD |
| cycles_completed | number | 已完成的 OKR 周期数 |

### 正文区块

| 区块 | 说明 | 更新方式 |
| --- | --- | --- |
| `## 专业知识` | 角色的专业判断力积累 | 可更新（新知识替换旧知识，或追加新条目） |
| `## 项目上下文` | 跨周期的项目理解 | 可更新（项目信息变化时覆盖） |
| `## 反思记录` | 从评分和回顾中提炼的教训 | 追加模式（每个 Cycle 一个三级标题，不删除历史记录） |

## 各角色记忆侧重

| 角色 | 专业知识侧重 | 项目上下文侧重 | 反思记忆侧重 |
| --- | --- | --- | --- |
| GM | 需求拆解模式、OKR 定义最佳实践 | 业务目标、客户反馈 | KR 定义质量、团队协作效率 |
| PD | 产品设计模式、用户体验原则 | 产品路线图、竞品分析 | 需求优先级判断准确度 |
| ArchD | 技术选型经验、架构模式 | 技术债务、性能基线 | 架构决策的长期效果 |
| BE/FE | 编码规范、工具链偏好 | 模块依赖关系 | 估时准确度、质量指标趋势 |
| QA | 测试策略、缺陷模式 | 高风险模块分布 | 测试覆盖有效性 |

## team.md 特殊说明

`team.md` 记录团队级别的组织智慧，不属于任何单一角色：

- **协作模式**：哪些角色组合协作效果好，哪些容易产生摩擦
- **流程改进**：交付幕的节奏是否合理，门禁是否过严/过松
- **跨周期趋势**：整体交付质量、协作效率的变化趋势

## 读写时机

### 写入时机：M4 阶段（Review Convergence）

当 `okr-review-score` 完成评分后，自动触发"经验提炼"：

1. **分析本次评分结果**：哪些 KR 得分高/低？为什么？
2. **提取可复用的经验**：什么做法有效？什么是陷阱？
3. **更新角色 wisdom 文件**：追加新的反思记录，更新专业知识
4. **更新团队 wisdom**：提炼团队级别的协作经验

### 读取时机：M0 阶段（Need Translation）

当 `okr-gm` 启动新的 OKR 周期时：

1. GM 读取自己的 wisdom，避免重复犯错
2. 在分配角色时，参考各角色的 wisdom 调整 KR 难度
3. 在定义 KR 时，参考历史数据设定合理的量化指标

### 读取时机：M2/M3 阶段（角色执行）

当 `okr-role-run` 启动角色执行时：

1. 角色读取自己的 wisdom，获取专业知识和历史教训
2. 基于历史经验选择更优的实现路径

## 更新规则

- wisdom 文件不存在时，由写入方自动创建（含 frontmatter 和空区块）
- 每次写入更新 frontmatter 的 `updated` 字段
- 完成一个完整 OKR 周期后，`cycles_completed` 加 1
- 反思记录按 Cycle 编号追加，不删除历史记录
- 专业知识和项目上下文可覆盖更新（新信息替换过时信息）
- 单个 wisdom 文件建议控制在 200 行以内，超出时精简早期反思记录（保留教训和改进，删除细节）
