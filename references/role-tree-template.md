# Role Tree Template

## 默认组织树

```text
GM 甲方总经理（甲方代理）
├── PD 产品总监
│   ├── PM 产品经理
│   ├── UI 设计师
│   └── TW 文档专家/DX
└── ArchD 技术总监
    ├── BE 后端开发工程师
    ├── FE 前端开发工程师
    ├── QA 测试工程师
    ├── DevOps 发布工程师
    └── SEC 安全工程师
```

## 角色职责表

| 角色 | 缩写 | KR ID 前缀 | 职责范围 | 上级 |
| --- | --- | --- | --- | --- |
| 甲方总经理 | GM | GM-KR | 甲方需求代理，定义 Objective 和顶层 KR，最终验收 | 甲方 |
| 产品总监 | PD | PD-KR | 产品方向管理，协调 PM、UI、TW 产出，确保产品方案完整 | GM |
| 产品经理 | PM | PM-KR | 需求分析、用户流程、验收标准、范围管理 | PD |
| UI 设计师 | UI | UI-KR | UI 设计稿、交互规范、视觉标准 | PD |
| 文档专家/DX | TW | TW-KR | API 文档、用户指南、CHANGELOG、开发者体验 | PD |
| 技术总监 | ArchD | ARCHD-KR | 技术方案选型、模块拆分、接口契约、非功能性约束 | GM |
| 后端开发工程师 | BE | BE-KR | API 实现、数据模型、业务逻辑、数据库变更 | ArchD |
| 前端开发工程师 | FE | FE-KR | 页面/组件实现、交互逻辑、样式与响应式 | ArchD |
| 测试工程师 | QA | QA-KR | 测试用例设计、自动化测试、缺陷验证、覆盖率 | ArchD |
| 发布工程师 | DevOps | DEVOPS-KR | CI/CD 流水线、部署脚本、环境配置、发布验证 | ArchD |

## KR ID 命名规范

格式：`{角色缩写}-KR{序号}`，例如：
- `PD-KR1`、`PD-KR2` — 产品总监的第 1、2 个 KR
- `PM-KR1`、`UI-KR1` — 产品经理、UI 设计师各自的第 1 个 KR
- `ARCHD-KR1` — 技术总监的第 1 个 KR
- `BE-KR1`、`FE-KR1` — 后端、前端各自的第 1 个 KR
- 序号从 1 开始，按创建顺序递增

## 上下级关系规则

- 下级角色的每个 KR 必须映射到上级的某个 KR（`上级 KR` 字段）
- 评分默认由直接上级给下级打分（GM→PD+ArchD, PD→PM+UI+TW, ArchD→BE+FE+QA+DevOps+SEC）
- GM 的 KR 由甲方（用户）最终确认
- 不允许跨级映射（BE-KR 不能直接映射到 GM-KR，必须经过 ARCHD-KR）

## 角色裁剪决策树

根据需求复杂度决定启用哪些角色：

```text
需求是否涉及 UI/页面？
├── 是 → 启用 FE
└── 否 → 跳过 FE

需求是否涉及后端逻辑/API/数据库？
├── 是 → 启用 BE
└── 否 → 跳过 BE

需求是否需要部署或环境变更？
├── 是 → 启用 DevOps
└── 否 → 跳过 DevOps

需求是否涉及认证、授权或敏感数据？
├── 是 → 启用 SEC
└── 否 → 跳过 SEC

需求是否需要对外文档或 API 文档更新？
├── 是 → 启用 TW
└── 否 → 跳过 TW

需求复杂度是否需要独立架构设计？
├── 是 → 启用 ArchD
└── 否 → ArchD 职责合并到 PD 或 BE
```

### 裁剪原则

- **最小角色集**：只启用需求真正涉及的角色，减少空转
- **合并而非删除**：跳过的角色职责应合并到最近的上级或同级角色
- **风险标注**：合并角色时必须说明被合并职责和潜在风险
- PD、QA 为默认启用角色（除非需求极其简单，如纯文档修改）

## 裁剪说明模板

| 角色 | 状态 | 说明 |
| --- | --- | --- |
| FE | 跳过 | 本次需求为纯后端 API，无 UI 变更 |
| SEC | 合并至 BE | 涉及简单鉴权，由后端开发工程师一并处理 |
| DevOps | 跳过 | 使用现有 CI/CD，无环境变更 |
